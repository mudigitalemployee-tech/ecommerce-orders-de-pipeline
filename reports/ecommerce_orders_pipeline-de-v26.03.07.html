Use Cases

Structured use cases derived from BRD/FRD context — actors, flows, pre/postconditions.

UC-01 — Automated Data Ingestion

CRITICAL   Actor: Pipeline Scheduler (Airflow)

The system automatically ingests raw data from source on a scheduled basis (daily) without manual steps.

AttributeDetail
PreconditionsSource system accessible. Credentials valid. Schema stable.
PostconditionsBronze layer contains full-fidelity copy of source records for this run.

Flow Steps:

  1. Airflow DAG triggers on schedule
  2. Pipeline connects and reads new/changed records
  3. Raw data written to Bronze layer unchanged
  4. Ingestion metadata logged
  5. Success notification sent

UC-02 — Data Transformation & Quality Enforcement

CRITICAL   Actor: Transform Engine

Cleaned, validated Silver layer produced from Bronze with DQ checks enforced and PII masked.

AttributeDetail
PreconditionsBronze layer populated by UC-01.
PostconditionsSilver layer contains clean, typed, deduped, PII-masked records with DQ score ≥ threshold.

Flow Steps:

  1. Read from Bronze layer
  2. Apply column normalisation (rename, type cast, date format)
  3. Remove duplicates
  4. Apply null handling rules
  5. Mask PII columns (SHA256)
  6. Run DQ checks
  7. DQ FAIL → halt, alert, quarantine
  8. DQ PASS → write to Silver

UC-03 — Business Analytics Consumption

HIGH   Actor: BI Tool / Dashboard / Analyst

Retail analysts consume Gold-layer aggregates — revenue by category, store performance, inventory KPIs.

AttributeDetail
PreconditionsSilver layer DQ-validated. Gold aggregation job completed.
PostconditionsStakeholders have access to up-to-date pre-aggregated metrics within SLA window.

Flow Steps:

  1. Gold aggregation job runs after Silver load
  2. KPI metrics calculated and written to Gold tables
  3. BI tool refreshes from Gold layer
  4. Analyst queries Gold for ad-hoc analysis
  5. Automated alert if KPI threshold breached

UC-04 — Pipeline Failure Detection & Alerting

HIGH   Actor: Monitoring System / On-Call Engineer

When any stage fails, the system detects, halts, and alerts the on-call owner with actionable details.

AttributeDetail
PreconditionsMonitoring configured. Alert channel set.
PostconditionsFailure visible within SLA. No silent failures. Bad data quarantined.

Flow Steps:

  1. Stage fails or DQ score below threshold
  2. Airflow marks task FAILED
  3. Alert sent to on-call channel
  4. Alert contains: DAG ID, failed task, error, row count
  5. Engineer triages and decides action
  6. Incident logged for post-mortem

UC-05 — Historical Data Backfill & Reprocessing

MEDIUM   Actor: Data Engineer

When business logic changes or a source issue is discovered, pipeline supports backfilling date ranges.

AttributeDetail
PreconditionsBronze layer retains raw data. Airflow supports backfill.
PostconditionsHistorical data reflects updated logic. Downstream reports corrected.

Flow Steps:

  1. Engineer identifies backfill window
  2. Airflow triggered with backfill parameters
  3. Bronze re-reads source for range
  4. Silver re-applies current transform logic
  5. Gold re-aggregates affected periods
  6. DQ checks on reprocessed data
  7. Stakeholders notified
